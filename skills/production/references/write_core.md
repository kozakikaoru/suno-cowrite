# 02 Style 組み立て — 記述子を層で設計する

Style 欄を「思いついたタグの列挙」ではなく「役割の決まった層の集合」として組むための必読資料。語彙辞典 **style-vocab (実効パスは呼び出しプロンプト参照)** と対で使う。並び順・タグ数・文字数上限などの基本規約と正確な数値は suno-spec §5 (実効版) が正で、本資料はその上に載る設計手順を扱う。

## 1. レイヤー法 — 基本形の語数配分

Style は次の層で組む。**合計はフル版で 10 タグ前後** (上限運用は suno-spec §5)。

| 層 | 語数 | 役割 | 例 (自作) |
|---|---|---|---|
| ジャンル | 1 (サブジャンルまで特定して 1 つ) | 最重要。**必ず先頭** | `japanese city pop` |
| 年代/文脈 | 1 | 録音・処理の慣習一式を引き寄せる | `80s` |
| ムード | 2 | 感情の方向。generic 語 (happy/sad) を避け特異的な語に | `nostalgic, bittersweet` |
| 楽器 | 2〜3 | 低音 → ドラム → 上物の順で決める (01 §5) | `funky slap bass, shimmering electric piano` |
| ボーカル | 1〜2 | 温度/力感/距離 (01 §4) | `smooth female vocals` |
| プロダクション | 1〜2 | 仕上げと空間の軸 (01 §6) | `analog tape warmth` |
| 非音楽語 (情景/エネルギー) | 1〜2 | 場面・動きの複合ショートカット (§3) | `night drive feel` |

- **先頭の語ほど強く効く** (プロンプトは高次属性に圧縮され、先頭・高頻度語が支配的 — 確度 Med)。飾りを先頭に置かない
- 全層を埋める義務はない。**枠が余っても足さない** (語数を盛るほど平均化して個性が消える)
- テンポを書くなら BPM 単独ではなくエネルギー語との併用が安定 (表記は suno-spec §5 の `bpm feel` 形式)

## 2. 語彙辞典 (style-vocab) の引き方

- **通読しない**。今回の曲で埋めたい枠 (ムード / 情景 / 質感 / エネルギー / 日本文化語) に該当するカテゴリだけ拾い読みする
- **確度 High / Med の語を骨格に使う**。辞典の確度列がそのまま採用基準になる
- **確度 Low・未検証の語は「実験枠」1 語まで**。使ったら【判断メモ】に「実験枠: 〇〇 (確度 Low)」と明記し、P が結果で検証できるようにする (実測結果は update-spec 経由で辞典に還元される)
- 辞典に無い語を発明したときも実験枠として数え、判断メモに書く (次回調査の候補になる)
- **日本語の意図語は英語に訳して使うのが原則** (日本語直書きは認識が不安定)。訳し方は §3 の手順で
- 辞典の「効かない・危険語」カテゴリは Style を確定する前に一度照合する

## 3. 抽象要望の翻訳 — 「疾走感」「エモく」を英語記述子に落とす

P・ブリーフの日本語の意図語を英語 Style に変換する標準手順。勘で英訳せず辞典を経由する。

1. **抜き出す** — ブリーフ・P の発言から意図語を拾う (「疾走感」「エモく」「チルに」「夜っぽく」)
2. **分解する** — その語が指すものを 4 枠に仕分ける: 感情 (ムード) / 動き・速度 (エネルギー) / 場面 (情景) / 手触り (質感)。「エモく」のような多義語は 2 枠以上に割れる
3. **辞典で置換する** — style-vocab の該当カテゴリから確度 High/Med の語を選ぶ。1 意図語 = 1〜2 記述子まで
4. **矛盾検査に回す** — 既存の層 (ジャンル・楽器・年代) と §4 で照合する
5. **対応を記録する** — 【判断メモ】に「疾走感 → driving (辞典 Med)」の形で 1 行残す (P が聴感で答え合わせできる)

変換例 (語の最終確度は辞典側が正):

| 意図語 | 分解 | 変換例 |
|---|---|---|
| 疾走感 | 動き | `driving` を第一候補に、`fast-paced` やテンポ帯を併記 |
| エモく | 感情 + 動き | `bittersweet` + `soaring` など分解して 2 語 (generic な `emotional` 単独にしない) |
| チルに | 動き + 質感 | `laid-back, mellow` (+ 場面を足すなら情景 1 フレーズ) |
| 夜っぽく | 場面 | `night drive` 系の情景フレーズ 1 つ |
| 切なく | 感情 | `melancholic` または `wistful` (静的なら後者) |
| 派手に | 密度 | `maximalist, glossy production` |

- **情景フレーズは 1 曲 1 つまで**。ジャンル等の主柱を立てた後ろに置く (情景だけの長文は圧縮で切り捨てられる)
- 翻訳しても方向が曖昧な要望 (「いい感じに」) は、翻訳せずマネージャー経由で具体化を求める

## 4. 相性と矛盾の検査

組み上がった Style を確定する前に、競合タグの類型で総当たりする。

| 矛盾の類型 | 例 | 扱い |
|---|---|---|
| 空間 | `reverb-heavy` × `lo-fi` / `dry` | どちらかに決める (01 §6) |
| 温度 | `warm` × `cold, icy` | どちらかに決める |
| エネルギー | `calm` × `aggressive` | 原則同居禁止。意図的な緊張ペアは `warm but lonely` のように **but で 1 フレーズ化して 1 枠扱い** (確度 Low〜Med、実験枠として判断メモに書く) |
| 密度 | `minimal, sparse` × `lush, maximalist` | どちらかに決める |
| 年代 | `80s` × `modern` | 年代は 1 つ (意図的な再解釈は 09 プレイブック) |
| ジャンル多重 | サブジャンル 3 つ以上の併記 | 主 1 + 修飾 1 まで。それ以上は平均化する |

- **タグ数と Style Influence は連動して決める**: タグを絞るほど追従度を高くできる (高追従 × 少数タグが定石。数値の定石帯は suno-spec §3 が正)。タグ 10 超のまま高追従にすると競合して濁る
- 検査手順: 各層を 1 回ずつ読み直し、(a) 同じ枠に 3 語以上入っていないか、(b) 上の類型に当たるペアがないか、(c) 歌詞の内容・物語が混入していないか (Style は音響指示専用)、を見る

## 5. Exclude 設計の定石

否定は Style 欄に書かず Exclude Styles へ (Style 欄の否定形は不安定 — suno-spec §5)。

- **少数精鋭**: 「この曲で混入すると事故る要素」だけを入れる (5 項目以内目安)。心配性で並べると生成の幅を不必要に削る
- **「事故ったら足す」運用**: 初回から埋め尽くさず、テイクで混入した要素を修正時に追加するのが基本
- 定石の類型: ジャンル汚染防止 (バラードに `edm, dubstep`) / 楽器事故 (アコースティック曲に `heavy distortion`) / ボーカル事故 (インスト曲に `vocals` — トグル・タグとの三重指定の一部) / ムード反転防止 (ダーク曲に `upbeat, cheerful`)
- Style 側と同じ語を両方に書かない (Style に `acoustic` + Exclude に `electronic` は可。Style の語を Exclude に入れるのは矛盾)

## 6. Persona 適用時の差分設計

前提 (suno-spec §6): Persona = アーティストの同一性の錨 / Style 欄 = 曲ごとのプロデューサー指示。登録済みアーティストでは 2 変種 (適用時 / フル) を納品する。

- **適用時 Style = フル版からアイデンティティ層を引き算したもの**: ジャンル帯・基準ボーカル像 (artist.yaml の genre_band / vocal 由来) は Persona が担うため書かない。残すのは曲固有の層 = ムード / 情景 / セクション演出寄りの楽器追加 / 今回だけの質感 (語数の目安は suno-spec §6 と制作 (songsmith) 定義に従う)
- **検査 2 点**: (a) 適用時 Style に genre_band / vocal 由来の語が残っていないか (残すと Persona と競合して崩れる)、(b) フル版と適用時版の曲固有層が同じ方向を向いているか (2 変種は「同じ曲の 2 通りの入稿」であって別の曲にしない)
- 基準ジャンル帯から遠い曲は Persona 自体が崩れやすい — 適用可否の判断枠組みは制作 (songsmith) 定義 (agents/songsmith.md) の作曲工程の規則に従う

## 7. コンペ 2 案の差軸カタログ

コンペモード (デモ A / B) では、**差は 1 軸だけで作る**。2 軸以上動かすと P がどの違いを聴いているのか分からなくなる。どの軸でもブリーフのテーマと genre_band の範囲内に収める。

| 差軸 | A 案 ⇔ B 案の例 |
|---|---|
| 年代軸 | 同じ曲想を `80s` 解釈 ⇔ `modern` 解釈 |
| 密度軸 | `sparse, intimate` (小編成) ⇔ `lush, layered` (壁) |
| ボーカル温度軸 | `breathy, intimate vocals` ⇔ `powerful soaring vocals` |
| 情景軸 | 同じムードを別の場面語で (`night drive` ⇔ `rainy day cafe`) |
| 動力軸 | 生バンド感 (`live drums, organic`) ⇔ 打ち込み (`electronic beat, synth-driven`) |

【コンペ提案】の冒頭に「今回の差軸: 〇〇軸」を 1 行で書き、各案の狙いを対にして示す。

## 8. 推敲チェックリスト

- [ ] 層の配分が §1 の基本形に収まっている (フル 10 タグ前後 / Persona 適用時は曲固有層のみ)
- [ ] ジャンルが先頭にあり、サブジャンルまで特定されている
- [ ] 非音楽語は確度 High/Med から 1〜2 語。Low・未検証・自作語は実験枠 1 語まで + 判断メモに明記した
- [ ] 情景フレーズは 1 つまでで、主柱の後ろに置いた
- [ ] 日本語の意図語がすべて英語化され、対応が判断メモに残っている (§3)
- [ ] §4 の矛盾類型で総当たりした (空間 / 温度 / エネルギー / 密度 / 年代 / ジャンル多重)
- [ ] 否定形・歌詞内容・実在アーティスト名が Style に混入していない (否定は Exclude へ)
- [ ] Exclude は少数精鋭で、Style と同語の矛盾がない
- [ ] タグ数と Style Influence の組み合わせが定石 (絞る × 高追従) に沿っている
- [ ] 文字数上限・タグ語彙・スライダー数値を suno-spec (実効版) で確認した

# 01 曲構成理論 — 器を先に決める

歌詞の失敗の多くは行の質ではなく、器 (構成) の選び間違いで起きる。この資料は、構成の型を選び、Suno のメタタグ列に写像し、歌詞量を決めるまでの手順。ワークフロー段階 1 (設計) で使う。

## 1. 基本 3 型と使い分け

### Verse–Chorus 型 (現代ポップスの標準)

- 形: Intro → Verse 1 → (Pre-Chorus) → Chorus → Verse 2 → (Pre-Chorus) → Chorus → Bridge → Chorus → Outro
- 性質: サビ (Chorus) が曲の顔。反復で刷り込む
- 選ぶ条件: フック勝負の曲、シングル志向、ショート動画展開の想定。**迷ったらこれ**

### AABA 型 (スタンダード/バラード系)

- 形: A → A → B → A。サビという概念がなく、A 自体が主役
- 性質: 物語が静かに進む。B で一度だけ視点が変わる
- 選ぶ条件: 語り中心のバラード、ジャズ・歌謡曲的な質感、サビの爆発が世界観に合わない曲
- Suno 上の注意: `[Chorus]` を使わず、A を `[Verse 1]` `[Verse 2]`… で並べ、B は `[Bridge]`。A 間のメロディの統一感は、行数とモーラ数を A 同士で揃えることで担保する (番号付きタグは別展開に振れやすいため。§3)

### EDM 型 (Build–Drop)

- 形: Intro → Verse → Build → Drop → Verse → Build → Drop → (Bridge) → Drop
- 性質: 感情のピークは歌詞ではなく Drop (音) が担う。歌詞は Build までの助走
- 選ぶ条件: ダンス/エレクトロ系。Drop はボーカルなし、または短いチャントのみ
- Suno 上の注意: `[Build]` `[Drop]` タグを使う。Drop 区間には歌詞を書かず、`[Drop]` 単独か、チャント 1 行 (02 §5) だけを置く

### 選択フロー

1. ブリーフに構成案があればそれに従う (作詞家が勝手に変えない。変えたい理由があれば納品時に提案として書く)
2. なければ: ダンス主体 → EDM 型 / サビで開く曲 → Verse–Chorus 型 / 静かな物語 → AABA 型
3. 型は混ぜない。「AABA だがサビもある」のような折衷は、Suno がどちらの構成にも寄れず崩れやすい。迷ったら Verse–Chorus 型に寄せ切る

## 2. J-POP 定型と英米ポップの対応表

| J-POP 用語 | 英米ポップ用語 | 役割 |
|---|---|---|
| イントロ | Intro | 世界に入る導入 |
| A メロ | Verse | 状況・風景の提示 |
| B メロ | Pre-Chorus | サビへの傾斜。緊張を上げる |
| サビ | Chorus | 感情と主題のピーク。タイトル/フックの置き場 |
| 間奏 | Interlude / Instrumental Break | 呼吸。場面転換 |
| C メロ | Bridge | 一度だけの視点転換・飛躍 |
| 落ちサビ | Stripped chorus (Final Chorus の前段) | 静めてから最後を最大化する助走 |
| 大サビ / ラスサビ | Final Chorus | 最後の最大サビ |
| アウトロ | Outro | 余韻 |

J-POP 定型のフル形: A → B → サビ → A' → B' → サビ → C (または落ちサビ) → 大サビ → アウトロ。

## 3. Suno メタタグへの写像表

構成表のセクション列を、そのままタグ列に変換する。タグは英語で、**suno-spec の語彙一覧にあるものだけ**使う。

| 構成表の書き方 | Suno タグ列 |
|---|---|
| イントロ (歌なし) | `[Intro]` (歌詞行を置かない) |
| A メロ 1 / 2 | `[Verse 1]` / `[Verse 2]` |
| B メロ | `[Pre-Chorus]` |
| サビ | `[Chorus]` |
| サビ後の畳みかけ | `[Post-Chorus]` |
| 間奏 | `[Instrumental Break]` (楽器を立てるなら `[Guitar Solo]` 等) |
| C メロ | `[Bridge]` |
| 落ちサビ | `[Bridge: stripped down, quiet]` + 直後に `[Chorus]`、または `[Chorus: stripped down, piano only, soft vocals]` |
| 大サビ | `[Chorus]` (持ち上げるなら直前に `[Key Change]`、または `[Chorus: full band, soaring vocals]`) |
| ラップ区間 | `[Rap]` (03 §5) |
| 語り | `[Spoken Word]` |
| アウトロ | `[Outro]` (フェードで閉じるなら `[Fade Out]` 併用、確実に終止させるなら `[End]`) |

### タグ運用の原則 (Suno 特性)

- **タグを置かないと歌詞全体が一塊として処理され、構成が崩れやすい**。全セクションに必ずタグを置く。
- **番号付きタグは「別展開」の指示**: `[Verse 1]` と `[Verse 2]` は異なるメロディ展開に振れやすい。1 番と 2 番の歌詞内容も変える (05 §3) のが噛み合う。
- **同一歌詞の `[Chorus]` 反復は「同じメロディ」の指示**: サビのメロディを固定したいなら、サビ歌詞は全サビで一字一句同じにするのが最も確実。サビごとに歌詞を変えるのはメロディも変わるリスクを取る行為 (妥協点は 05 §5 の 1 語差し替え)。
- **パラメータ付きタグ `[Chorus: stripped down, piano only]` はセクション単位の演出指示**。Style 欄を変えずに音を動かせる、実用上最も強力な制御手段。感情曲線との連動は 05 §6。
- タグが稀に歌われる事故は確率的でゼロにできない。事故が出たら該当タグを標準語彙内の別表現に変えて再生成する (06 第 3 部)。

### タグ列の実例 (1 曲分のスケルトン)

§5 の構成表をタグ列に落とすと次の形になる。設計段階でこのスケルトンまで作り、歌詞は空欄のまま執筆段階に渡す。

```text
[Intro]

[Verse 1]
(6 行)

[Pre-Chorus]
(4 行)

[Chorus]
(6 行)

[Verse 2]
(6 行)

[Pre-Chorus]
(4 行)

[Chorus]
(1 回目と同一歌詞)

[Bridge: stripped down, vulnerable vocals]
(4 行)

[Chorus: full band, soaring vocals]
(末行のみ 1 語差し替え)

[Outro]

[End]
```

## 4. テンポ別・曲長別の歌詞量目安 (日本語)

**詰め込みは早口化を、不足は早終了を招く** (Suno は歌詞量と行密度から曲の尺を推定する)。器に合った量を設計段階で決め、執筆で守る。

| テンポ帯 | 1 行の目安 (モーラ) | セクション行数の目安 | 3〜4 分想定の本文総量 (タグ除く) |
|---|---|---|---|
| スロー (〜80 BPM 感) | 6〜10 | Verse 4〜6 行 / Chorus 4〜6 行 | 250〜450 字 |
| ミッド (80〜115 BPM 感) | 8〜12 | Verse 6〜8 行 / Chorus 6〜8 行 | 400〜700 字 |
| アップ (115 BPM 感〜) | 10〜16 | Verse 8 行前後 / Chorus 8 行前後 | 550〜900 字 |

- Pre-Chorus は 2〜4 行、Bridge は 2〜6 行が目安 (どのテンポ帯でも)。
- 上限値は suno-spec が正 (現行スナップショット: 上限 5,000 字・実用 3,000 字以下推奨)。上表の範囲なら大きく余裕がある。3,000 字に近づくのは長尺曲か詰め込みのどちらかで、たいてい後者。
- 曲長の調整:
  - 短くしたい (2 分台) → 削る順は「2 番の Pre-Chorus → Post-Chorus → 間奏 → 2 番 Verse の後半」。**サビは削らない**
  - 長くしたい (5 分前後) → 間奏・Post-Chorus・Bridge を足す。Verse を 3 つにするより反復と間奏で伸ばす方が破綻しない
- 配分のよくある失敗:
  - サビだけ行数が多く Verse が薄い → サビ前で息切れした印象になる。行数配分は構成表で先に固定する
  - 全セクションが同じ行数 → 呼吸がなく単調になる。Pre-Chorus と Bridge は短くして緩急を作る
- 英語はモーラではなくストレス数で数える (07 §4)。行数の目安は同じ。

## 5. 構成表テンプレート (設計の成果物)

設計段階で次の表を作ってから執筆に入る。例はミッドテンポ・V 字型の想定。

| # | セクション | タグ | 行数 | 1 行モーラ | 感情強度 (05) | 役割・モチーフ (04) |
|---|---|---|---|---|---|---|
| 1 | イントロ | `[Intro]` | 0 | — | 2 | 楽器のみ |
| 2 | A メロ 1 | `[Verse 1]` | 6 | 8〜10 | 3 | 夜の街の風景提示 |
| 3 | B メロ | `[Pre-Chorus]` | 4 | 8 | 5 | 疑問形で傾斜 |
| 4 | サビ | `[Chorus]` | 6 | 10〜12 | 8 | タイトル回収・フック |
| 5 | A メロ 2 | `[Verse 2]` | 6 | 8〜10 | 4 | 時間経過後の風景 |
| 6 | B メロ 2 | `[Pre-Chorus]` | 4 | 8 | 6 | 決意の芽 |
| 7 | サビ | `[Chorus]` | 6 | 10〜12 | 8 | 同一歌詞で反復 |
| 8 | C メロ | `[Bridge: stripped down, vulnerable vocals]` | 4 | 6〜8 | 3 | 反転・告白 |
| 9 | 大サビ | `[Chorus]` | 6 | 10〜12 | 10 | 末行のみ 1 語差し替え |

- [ ] 感情強度が平坦になっていない (05 §1 のルール)
- [ ] 1 回目のサビが曲頭から 60〜90 秒相当 (イントロ + Verse + Pre-Chorus が長すぎない) に来る (02 §6)

# 05 感情曲線 — 曲は感情の設計図で聴かせる

感情曲線 = 曲頭から曲末までの感情の強度と質の軌跡。歌詞・構成・サウンド演出をこの 1 本に従わせる。ワークフロー段階 1 (設計) で曲線を決め、段階 3 (推敲) でずれを検査する。

## 1. 曲線の書き方 (設計の成果物)

各セクションに**感情強度 1〜10** と**感情の質 (1 語)** を割り付け、構成表 (01 §5) に記入する。

例 (V 字型。自作の想定曲「夜光性メロディ」):

| セクション | 強度 | 質 |
|---|---|---|
| `[Intro]` | 2 | 予感 |
| `[Verse 1]` | 3 | 孤独 |
| `[Pre-Chorus]` | 5 | 焦り |
| `[Chorus]` | 8 | 切望 |
| `[Verse 2]` | 4 | 回想 |
| `[Pre-Chorus]` | 6 | 決意の芽 |
| `[Chorus]` | 8 | 切望 |
| `[Bridge]` | 3 | 静けさ (どん底) |
| `[Chorus]` (大サビ) | 10 | 解放 |
| `[Outro]` | 4 | 余韻 |

ルール:

- **強度 10 は 1 曲に 1 箇所だけ** (たいてい大サビ)。2 箇所あるとどちらも 10 に聞こえない。
- 同じ強度が 3 セクション続いたら平坦。どこかを上げ下げする。
- 隣接セクションの強度差 5 超の跳躍は 1 曲 1 回まで (Bridge → 大サビの専売)。

「質」の語彙が出ないときは次から選ぶ: 予感 / 孤独 / 焦り / 憧れ / 切望 / 回想 / 決意 / 静けさ / 強がり / 安堵 / 解放 / 祝祭 / 余韻。

## 2. 基本パターンと選び方

| パターン | 形 | 合う題材 |
|---|---|---|
| **上昇型** | 3 → 5 → 7 → 10 と積み上げる | 応援歌、旅立ち、始まりの曲 |
| **V 字型** | 中盤 (Bridge) で一度落としてから最大へ | 喪失からの再生、葛藤もの。最も汎用 |
| **静→爆発型** | 前半を 2〜4 で抑制し、後半で一気に解放 | 抑圧からの解放、告白もの |
| **余韻型 (下降)** | ピークを 2 回目サビに置き、後半は静かに閉じる | 別れ、追憶、AABA 型バラード |

選び方: ブリーフに感情曲線の指定があれば従う。なければ題材で選び、迷ったら V 字型。

## 3. 1 番と 2 番の情報差

2 番が 1 番の言い換えだけの曲は、2 番で聴くのをやめられる。**2 番には新情報を 2 つ以上**入れる。

情報差の作り方 (どれか 1 つを軸にする):

| 軸 | 1 番 | 2 番 |
|---|---|---|
| 時間経過 | 出会った夜 | 三ヶ月後の朝 |
| 視点転換 | 私から見た君 | 君から見えていたはずの私 |
| 内面 → 外面 | 心の中の独白 | 実際に交わした会話・行動 |
| 抽象 → 具体 | 予感・気配 | 決定的な出来事 |

- 検査法: **2 番だけを読んで、1 番にない事実・場面・転回を 2 つ挙げられるか**。
- Suno 特性: `[Verse 1]` / `[Verse 2]` の番号付きタグは別のメロディ展開を誘導する (01 §3)。歌詞側も別内容にするのが噛み合う。

実例 (時間経過軸。自作)。1 番 A メロの抜粋:

```text
改札で 手を振った
帰り道は まだ二人分
```

2 番 A メロの抜粋:

```text
おなじ改札 通り抜ける
定期券だけ 更新されてく
```

同じ場所 (改札) を再訪させると時間経過が 1 行で伝わり、モチーフの回収 (04 §2) も兼ねられる。新情報は「一人になった」「季節が進んだ」の 2 つ。

## 4. ブリッジ → 大サビのカタルシス設計

カタルシスは「落差」で作る。手順:

1. **Bridge で曲中最低クラスの強度まで一度落とす** (V 字型なら 2〜3)。音数も減らす: 行を短く・少なく (2〜4 行)
2. Bridge の内容は次のどれか 1 つに絞る: (a) 最も個人的な告白、(b) 時間・視点の飛躍、(c) それまでの前提の反転 (04 §2)
3. 落ちサビを挟むなら、同一のサビ歌詞を `[Chorus: stripped down, quiet vocals]` で静かに歌わせ、続く大サビとの落差を最大化する
4. **大サビで最大強度**。ここだけに許される演出: `[Key Change]`、サビ歌詞の 1 語差し替え (§5)

## 5. 大サビでの歌詞変化 (1 語差し替えの技法)

Suno は同一歌詞の `[Chorus]` 反復でメロディを固定する (01 §3)。大サビで歌詞を大きく変えるとメロディごと変わるリスクがある。両立の妥協点:

- **サビ歌詞の 1 箇所 (1 語〜1 行) だけを差し替え、残りは一字一句同じにする**
- 差し替え位置はサビ末行が安全 (頭を変えるとフックが揺れる。02 §2)
- 差し替えの定石: 否定 → 肯定 / 疑問 → 断言 / 過去 → 現在
- 例 (自作): 通常サビ末「まだ言えないまま」→ 大サビ末「もう言える気がした」(モーラ数はほぼ維持する)

### 同一歌詞のまま感情を変える 2 つの道具

サビ歌詞を固定しても (01 §3)、聞こえ方は変えられる。

1. **直前の行で文脈を変える** — 2 番サビの直前 (Pre-Chorus 末) に反転の 1 行を置くと、同じサビが別の意味で響く (1 番では願望だった行が、2 番では強がりに聞こえる、など)
2. **演出タグで歌わせ方を変える** — 落ちサビ `[Chorus: restrained, almost whisper]` → 大サビ `[Chorus: full band, soaring vocals]`。歌詞は同一のまま強度だけを動かせる (§6)

## 6. 感情とサウンドの連動 (パラメータ付きタグへの翻訳)

感情曲線は歌詞だけでなく音にも指示する。**強度と質をタグ修飾子に翻訳**し、該当セクションのタグに付ける (タグ本体は suno-spec の語彙、修飾子は短い英語の形容)。

| 曲線上の状態 | タグ修飾の例 |
|---|---|
| 低強度・内省 (2〜3) | `[Verse 1: soft vocals, minimal arrangement]` |
| 傾斜・緊張 (5〜6) | `[Pre-Chorus: building intensity]` |
| ピーク (8) | `[Chorus: full band, soaring vocals]` |
| どん底・無防備 (Bridge) | `[Bridge: stripped down, piano only, vulnerable vocals]` |
| 最大 (10) | `[Chorus: explosive, layered harmonies]` |
| 余韻 | `[Outro: fading, gentle]` |

- 修飾子は 1 タグに 1〜3 個まで。**全セクションに付けると効きが薄まる** — 曲線の要所 (どん底・最大・転換点) だけに付ける。
- composer も同じブリーフの感情曲線から Style と演出を設計する。作詞家のタグ修飾は「セクション単位の感情演出」に限り、ジャンル・楽器編成の全体指定 (Style 欄の領分) を書かない。

## 7. 推敲チェックリスト

- [ ] 強度 10 が 1 箇所だけにある
- [ ] 同じ強度が 3 セクション以上続いていない
- [ ] 2 番だけ読んで、新情報を 2 つ挙げられる
- [ ] Bridge が「落とす」役割を果たしている (行数・音数も減っている)
- [ ] 大サビの歌詞変化が 1 箇所以内に収まっている (Suno のメロディ固定と両立)
- [ ] タグ修飾子が要所だけに付いており、Style 欄の領分を侵していない
- [ ] 各セクションの歌詞のトーンが、割り付けた「感情の質」と一致している

# 06 日本語作詞 — 歌える日本語と Suno 入稿変換 (完全版)

この資料は 3 部構成。**第 1 部 = 日本語で「歌える」歌詞を書く技術** (作品版の質)、**第 2 部 = 作品版 → 入稿版の機械的変換規則** (Suno の発音事故対策)、**第 3 部 = 生成後のガチャ運用**。表現の質は作品版で追い、発音対策は入稿版で吸収する — この 2 つを混ぜない。

- **作品版**: 漢字かな交じり。P への提示・YouTube 概要欄・字幕に使う
- **入稿版**: ひらがな化・助詞ローマ字化・括弧排除済み。Suno の Lyrics 欄に貼る唯一の版

---

## 第 1 部: 日本語で「歌える」歌詞を書く

### 1. モーラ感覚と字余り

- 日本語の歌は音符 1 つにモーラ 1 つが基本 (数え方は 03 §4)。
- 字余り (割付 ±1〜2 モーラ) はメロディ側が細かい音符で吸収するので許容される。ただし Suno では**行が長いほど詰め込まれる**ため、迷ったら字余り方向ではなく字足らず方向に倒す方が事故が少ない。
- 助詞を 1 つ削ると 1 モーラ稼げる (§3)。音数合わせのための無意味な「ああ」「そう」の挿入は最後の手段。

### 2. 母音とサビ適性 (伸ばしどころの設計)

日本語は開音節言語で、**行末・フレーズ末の母音が伸ばされやすい**。サビの決め所は、伸ばして映える母音で終わる語を選ぶ。

| 行末の音 | 伸ばしたときの質感 | サビ末適性 |
|---|---|---|
| a 段 (〜だ、〜な、〜か) | 開放・明るい | ◎ |
| o 段 (〜よ、〜も、〜ろう) | 温かい・包む | ◎ |
| e 段 (〜ね、〜て) | 呼びかけ・切なさ | ○ |
| i 段 (〜い、〜み) | 締まる・鋭い | ○ (切実系の曲) |
| u 段 (〜う、〜く) | こもる | △ |
| ん / っ | 伸ばせない | × (サビ末では避ける) |

- 悪い例 → 良い例 (サビ末。自作): 「きみを待ってたんだけど」(末尾が「ど」で濁って重い) → 「きみを待ってた 今も」(o 段で開放して伸ばせる)。
- Verse の行末は多様でよい。**サビの最後の 1 音だけは必ず設計する**。

### 3. 口語/文語と助詞省略

- 基準はアーティストの口調 (character.md)。口語基調のアーティストに文語 (「〜せし」「〜ゆえに」) を混ぜると浮く。文語を使うのは世界観設定に根拠があるときだけ。
- **助詞省略は歌詞の標準技法**: 「街の灯りが滲む」→「街の灯り 滲む」。省略で生じた間 (字空け) がそのままブレスになる。
- 省略しすぎの兆候: 1 行に裸の名詞が 3 つ以上並ぶ (「夜 街 灯り 君」)。単語の羅列は AI 作詞にありがちな既視感パターンでもあるため避ける。
- 残すべき助詞: 意味の対立を担う助詞は削らない (「君**が**いない」と「君**は**いない」の差が立つ場面)。

### 4. 語尾のバリエーション (単調の回避)

- 同じ語尾が 3 行連続したら黄信号 (韻として設計した場合を除く。03 §2)。
- 語尾の持ち駒: 「〜た」(完了) / 「〜る」(現在) / 「〜て」(継続・余韻) / 「〜ない」(否定) / 体言止め / 依頼・命令 (「〜して」) / 疑問 (「〜の?」) / 呼びかけ (「〜よ」「〜ね」)。
- 悪い例 → 良い例 (自作):
  - 悪い: 「歩いた / 探した / 泣いた / 笑った」(「た」4 連続)
  - 良い: 「歩いた / 探してる / 泣かないよ / 笑ってみせる」(完了 → 現在 → 否定 + 呼びかけ → 意志)
- Pre-Chorus の末尾は「言い切らない形」(〜て、〜まま、体言止め) にするとサビへ傾斜する。

### 5. 作品版での英単語・カタカナ語

- 原則: **1 曲に英語フレーズの系統は 1 つまで** (サビの決めフレーズか、合いの手のどちらか)。あちこちに散らすと安っぽくなる。
- 「日本語で言える語をわざわざ英語にする」のは摩耗表現の代表格。使うなら日本語で置換できない理由を持つ。
- カタカナ語 (ネオン、ガラス、メロディ) は日本語の語彙として自由に使ってよい。入稿時の扱いは第 2 部 R6。
- チャント (oh oh / la la) は小文字ローマ字で書く (02 §5)。これは英語フレーズの系統には数えない。

## 第 2 部: Suno 入稿変換規則 (作品版 → 入稿版)

### 変換の三原則

1. **一方向**: 変換は常に「作品版 → 入稿版」。入稿版を直接編集しない (直したくなったら作品版を直して再変換する)
2. **機械的**: 変換中に表現判断をしない。下の規則表 R1〜R9 を番号順に適用するだけ
3. **行構造の保存**: 変換で行の数・順序・分割を変えない (行構造 = 譜割り。03 §4)

### 変換の実行手順 (毎回この 5 ステップ)

1. **読み確定メモ**: 作品版の多読み語・数字・英単語を列挙し、R1 / R5 / R6 の決定を先にメモする
2. **機械変換**: R1〜R9 を番号順に全行へ適用する (1 行ずつ。表現はいじらない)
3. **チェックリスト**: 下の変換チェックリストを全項目確認する
4. **逆読みテスト**: 入稿版だけを音読し、作品版と同じ歌詞に聞こえるか確かめる
5. **凍結**: 以後の修正は作品版に入れて 1 からやり直す (入稿版への直接パッチは禁止)

### 変換規則表 (R1〜R9、適用順)

| # | 規則 | 内容 |
|---|---|---|
| **R1** | 読みの確定 | 多読み語 (下の判断基準表) の読みを、作品版の意図どおりに確定する。判断基準: (1) 設計のモーラ割付に合う方 (「あす」= 2 モーラ / 「あした」= 3 モーラ)、(2) アーティストの口語/文語トーン (「ゆく」= 文語寄り / 「いく」= 口語)、(3) 迷ったら声に出して自然な方 |
| **R2** | 漢字の全ひらがな化 | R1 で確定した読みで、**漢字をすべてひらがなに開く** (固有名詞も含む)。カタカナ語はカタカナのまま残す (読みが一意のため)。同音異義の紛れは無視してよい (Suno は音だけを歌う) |
| **R3** | 助詞のローマ字化 | 係助詞「は」→ `wa`、格助詞「へ」→ `e`、格助詞「を」→ `wo`。**前後に半角スペース**を入れる (「きみ wo さがす」)。語の一部の は/へ/を (「はな」「へや」) は対象外。細則は下記 |
| **R4** | 括弧・ルビの全排除 | `()` `（）` `「」` `『』` `《》` と、タグ以外の `[]` をすべて除去する。**括弧はコーラス/リフレイン指示として誤解釈されうる** (Suno 仕様)。セリフの「」は外して地の文にする。ルビ・当て読み (「本気」と書いて「マジ」と読ませる類) は、読ませたい音の方だけを書く |
| **R5** | 数字の読み下し | 算用数字・漢数字を読みのひらがなに開く: 「2 人」→「ふたり」、「17 歳」→「じゅうななさい」、「100 万回」→「ひゃくまんかい」。英語で読ませたい数字だけ英語綴り (「seventeen」)。年号などの長い数字は読みを設計時に決めておく (「2049」→「にせんよんじゅうきゅう」) |
| **R6** | 英単語の扱い | 英語発音させたい語は**英語綴りのまま**残し、前後に半角スペースを置く (「ひかりの melody」)。日本語発音させたい外来語は**カタカナ**にする (「メロディ」)。頭字語 (DNA、SOS) は発音が不安定なのでカタカナ読みに開く (「ディーエヌエー」「エスオーエス」) |
| **R7** | 記号の整理 | 「…」「―」「〜」は除去する (音を伸ばしたい箇所は長音「ー」で書く: 「ずっとー」)。「!」「?」は残す場合は半角にする (歌唱への影響は軽微)。読点「、」句点「。」は除去し、間は字空けか行分割で表す |
| **R8** | スペースの整理 | 作品版の字空けは維持する (ブレス位置のヒント)。R3 / R6 で入れた半角スペースの連続は 1 つにまとめる。行頭・行末の余分なスペースは削る |
| **R9** | タグの保全 | 構成タグ・パラメータ付きタグ (`[Chorus: stripped down]` 等) は英語のまま一切変更しない。タグが suno-spec の語彙一覧にあることを最終確認する |

### R1 用: 多読み語の判断基準表 (代表例)

| 表記 | 読み候補 (モーラ数) | 選び方の目安 |
|---|---|---|
| 明日 | あす (2) / あした (3) | モーラ割付を優先。「あす」はやや文語、「あした」は口語 |
| 行く | いく (2) / ゆく (2) | 歌では「ゆく」が滑らかで伸ばしやすい。口語感なら「いく」 |
| 何 | なに (2) / なん (2) | 続く語で決まる (「なにを」「なんで」)。誤読されやすいので必ず開く |
| 二人 | ふたり (3) / ににん (3) | 歌詞ではほぼ「ふたり」。誤読防止のため必ず開く |
| 一人 | ひとり (3) / いちにん (4) | 同上。「ひとり」 |
| 今日 | きょう (2) / こんにち (4) | ほぼ「きょう」。開かないと誤読リスクが残る |
| 昨日 | きのう (3) / さくじつ (4) | ほぼ「きのう」 |
| 風 | かぜ (2) / ふう (2) | 単独なら「かぜ」。「あんな風に」は「あんなふうに」と開いて書き分ける |
| 空 | そら (2) / くう (2) / から (2) | 「そら」以外の意図なら特に注意して開く |
| 方 | ほう (2) / かた (2) | 文脈で割れる代表格。必ず開く |
| 辛い | つらい (3) / からい (3) | 文脈で判別不能な代表格。必ず開く |
| 間 | ま (1) / あいだ (3) | モーラ数が大きく変わるため割付と必ず照合する |
| 今 | いま (2) / こん | 単独なら「いま」。熟語は熟語ごと開く (「今夜」→「こんや」) |
| 家 | いえ (2) / うち (2) | 建物なら「いえ」、生活の場なら「うち」。口調設定とも照合する |
| 月 | つき (2) / がつ (2) / げつ (2) | 空の月は「つき」。日付は R5 と併せて開く (「三月」→「さんがつ」) |
| 後 | あと (2) / のち (2) / うしろ (3) | 三方向に割れる代表格。必ず開く |

原則: 表にない語でも、**読みが 2 つ以上ありうる漢字はすべて開けば安全**。R2 で全ひらがな化するため、この表は「どの読みで開くか」を決めるためだけに使う。

### R3 用: 助詞ローマ字化の細則

- 対象は**助詞として機能している は/へ/を だけ**:
  - 「わたし**は**」→「わたし wa」 (対象)
  - 「**は**な (花)」→「はな」のまま (語の一部なので対象外)
  - 「こんにち**は**」→ 挨拶語は分割せず、音のとおり「こんにちわ」に書き換える
- 複合形は「は」の部分だけ変換する: 「〜では」→「で wa」、「〜とは」→「と wa」、「〜へは」→「e wa」
- 「へ」は方向の格助詞のみ対象 (「きみのもと e」)。「へや (部屋)」等は対象外
- 「を」は目的格をすべて `wo` にする (「よる wo こえて」)
- スペース規則: ローマ字助詞の前後に半角スペース 1 つ。行末に来る場合は後ろのスペース不要

### 変換の全行例 (ビフォーアフター。自作)

作品版:

```text
[Verse 1]
街の灯り ほどけて
君を探す 夜の底
「まだ間に合う」と 呟いた
2 人分の 明日へ
```

入稿版:

```text
[Verse 1]
まちのあかり ほどけて
きみ wo さがす よるのそこ
まだまにあうと つぶやいた
ふたりぶんの あした e
```

適用された規則: R1 (明日 → 「あした」を選択 ※3 モーラ割付のため / 間に合う → 「まにあう」)、R2 (全ひらがな化)、R3 (を → wo、へ → e)、R4 (「」の除去)、R5 (2 人 → ふたり)。行数は 4 → 4 で不変 (三原則 3)。

もう 1 例 (数字・英単語・係助詞を含むサビ。自作):

作品版:

```text
[Chorus]
午前 2 時の melody
君へ 鳴らす 100 回目
今夜は 眠らないまま
「おやすみ」は 言わない
```

入稿版:

```text
[Chorus]
ごぜんにじの melody
きみ e ならす ひゃっかいめ
こんや wa ねむらないまま
おやすみ wa いわない
```

適用された規則: R3 (へ → e、係助詞 は → wa を 2 箇所)、R4 (「」の除去)、R5 (2 時 → にじ、100 回目 → ひゃっかいめ)、R6 (melody は英語発音のまま維持)。

### 変換チェックリスト (全項目必須)

- [ ] 漢字が 1 文字も残っていない (固有名詞を含む)
- [ ] 助詞の は/へ/を がすべて `wa` / `e` / `wo` になっている (語中の は/へ/を を誤変換していない)
- [ ] あらゆる括弧 `()` `（）` `「」` `『』` が残っていない (タグの `[]` を除く)
- [ ] 算用数字・漢数字が残っていない
- [ ] 「…」「〜」「―」が残っていない (伸ばしは「ー」)
- [ ] 英字の連続が「意図した英単語」だけになっている (頭字語はカタカナ化済み)
- [ ] 行数・行順が作品版と完全に一致している
- [ ] タグがすべて suno-spec の語彙一覧にある

- [ ] 逆読みテスト: 入稿版を音読すると作品版と同じ歌詞に聞こえる (表現を変えていない)
- [ ] 本文総字数が実用目安 (約 3,000 字) 以下 — 上限値の最新は suno-spec を確認

## 第 3 部: ガチャ運用の心得 (生成後の修正対応)

同一入力でも生成ごとの揺らぎが大きい。**1 回で完成させる前提を捨て、修正対象を 1 箇所ずつ潰す**。

### 症状別の対処表 (P からのフィードバック対応)

| 症状 (P の報告例) | 原因の見立て | 歌詞側の対処 |
|---|---|---|
| 「2 番 A メロが早口」 | 該当セクションの詰め込み | 該当セクションだけ総モーラを 2〜4 割減らす (行を削るか各行を短くする)。他のセクションは触らない |
| 「間延びする」「曲が早く終わる」 | 歌詞量の不足 | 該当セクションに行を足す (Verse 4 行 → 6 行など)。01 §4 の総量目安に近づける |
| 「読み方が変」 | 表記の解釈揺れ | 該当語だけ表記を変える: ひらがな → ローマ字 (「こころ」→「kokoro」)、字空けの追加、カタカナ化 |
| 「サビのメロディが毎回違う」 | サビ歌詞の不一致 | 全サビの歌詞を一字一句同一にする (大サビの差し替えは 1 箇所まで。05 §5) |
| 「構成が無視される」 | タグ不足・語彙外タグ | 全セクションのタグの有無と、suno-spec の語彙一覧への適合を確認する |
| 「タグが歌われた」 | 確率的な事故 | 該当タグを標準語彙内の別表現へ変える (再生成だけで直ることも多い) |
| 「歌詞の一部が飛ばされる」 | 行の詰め込み・記号の残存 | 該当行を分割し、変換チェックリストを再実行する |
| 「英単語が日本語なまりになる」 | 英語綴りの解釈揺れ | 前後の半角スペースを確認し、より一般的な綴りに変える。日本語読みでよいならカタカナへ (R6 の再適用) |
| 「助詞が『ハ』『ヘ』と歌われる」 | R3 の変換漏れ | 該当行の は/へ/を を確認して wa / e / wo 化し、チェックリストを再実行する |

### 運用ルール

- 修正は **1 回の再生成につき 1 箇所** (複数を同時に変えると、どの変更が効いたか分からなくなる)。
- 修正は必ず作品版側で行い、入稿版は再変換で作り直す (三原則 1)。
- 日本語曲は英語曲よりガチャ回数が嵩みやすい (仕上げまで数十生成という報告もある)。初回からテンポに合った歌詞量 (01 §4) を守ることが、回数を減らす最大の節約になる。
- モデル更新で日本語発音が改善しても、入稿変換を弱める判断は spec 更新 (update-spec) とセットで行う。**作品版は変換方針に関わらず不変** — これが二重管理の利点。

# 07 英語作詞 — 簡潔版 (落とし穴の回避に絞る)

英語歌詞はモデルの素の英語力で一定水準が出るため、この資料は**非ネイティブ発想の落とし穴回避と検査**に絞る (v1 簡潔版)。構成・フック・感情曲線・モチーフは 01 / 02 / 04 / 05 が言語共通でそのまま適用される。

## 1. 優先順位の原則

**意味の自然さ > リズム (ストレス) > 韻**。韻のために意味や語順を壊すのが最大の失敗パターン。迷ったら韻を捨てる。

## 2. 非ネイティブの落とし穴 (執筆時の禁止パターン)

| 落とし穴 | 悪い例 (自作) | 直し方 |
|---|---|---|
| 和製英語 | my boom / skinship / one-pattern days | 英語圏で通じる語に置換する (obsession / closeness / same old days) |
| 韻のための不自然な倒置 | a love so true I never knew の型を乱発 | 倒置は 1 曲 1 回まで。平叙の語順で書き直す |
| 直訳された比喩 | my chest is hot (胸が熱い) | 英語の身体感覚に翻訳する (my heart's on fire) |
| 冠詞・単複の事故 | walk on street | 歌詞でも a / the は原則保つ。省くのは口語の定型表現だけ |
| 抽象語の連打 | destiny・eternity・miracle が 1 曲に全部 | 具体物に置換する (04 §3 は英語でも適用) |
| 文中の大文字化 | Love、Dream を文中で大文字にする | 文頭以外は小文字にする (発音解釈への余計な影響も避けられる) |
| 冗長ペア | return back / fall down deep | 動詞に含まれる意味を重ねない (return / fall deep) |
| 直訳の呼びかけ | hey, my dear friend | 歌の呼びかけは you / babe / my love などの定型に寄せるか、2 人称で通す |

## 3. コロケーション検査の手順

不自然さの大半は動詞と名詞の組み合わせで起きる。

1. 執筆後、動詞 + 名詞の組をすべて抜き出す (make a promise / take a breath …)
2. 「日本語から直訳した組」に印を付ける (例: see a dream → 正しくは have a dream)
3. 印が付いた組は、その名詞に対する定番動詞へ置き換える。確信が持てない組は納品時の【設計メモ】に不安要素として明記する

## 4. ストレスとシラブル (行のリズム)

- 英語は強勢拍リズム。**行の長さはシラブル数ではなくストレス (強勢) の数で数える**。
- 目安: ポップスの標準は 1 行 3〜5 ストレス。対応する行同士 (対句・同メロディ想定の行) はストレス数を一致させる。
- 例 (自作): "I **walk** the **emp**ty **street** a**lone**" = 4 ストレス。対句の行も 4 ストレスで揃える。
- 多音節語 (beautiful、memories) は行頭・行中に置き、行末は 1〜2 シラブルの強い語で終えると締まる。
- Suno は行構造から譜割りを推定する (03 §4 と同じ)。1 行 6 ストレス超は分割を検討する。
- 縮約形は歌詞の標準文体であり、シラブル調整の道具でもある: I am → I'm (2 → 1 シラブル)、going to → gonna、want to → wanna。縮約しない形は強調として響く (意図を持って 1 曲 1 回まで)。

## 5. 韻 (rhyme) は自然さ優先

- 現代ポップスは perfect rhyme (time / climb) より **slant rhyme (home / alone、fire / higher) が主流**。無理な perfect より自然な slant を選ぶ。
- スキームは ABCB (2・4 行末だけ揃える) が最も融通が利く。全行 AABB は童謡調になりやすい。
- 韻計画の立て方は 03 §1・§3 と共通。母音だけ揃えるアソナンスは英語でも有効。

## 6. 入稿変換 (英語は原則そのまま)

英語歌詞は発音事故対策が不要のため、**入稿版 = 作品版と同一**とし、納品時にその旨を明記する。ただし次だけは適用する。

- 数字は綴りに開く: 17 → seventeen、2 AM → two a.m.
- 括弧 `()` は使わない (コーラス反復と誤解釈されうるのは日本語と同じ)。背景コーラスは `[Backing Vocals]` タグか行分割で表す
- 発音が不安定な頭字語は綴りを開くかタイトル側だけに留める (一般的な語感の頭字語はそのままでよい)
- タグは suno-spec の語彙一覧にあるものだけ使う

## 7. 推敲チェックリスト

- [ ] §2 の落とし穴 8 種に該当する行がない
- [ ] 直訳コロケーションの疑いが残る組は【設計メモ】に明記した
- [ ] 対応する行同士のストレス数が揃っている
- [ ] 倒置が 1 回以下に収まっている
- [ ] 韻のために意味が不自然になった行がない
- [ ] 数字が綴りに開かれ、括弧が残っていない

# 08 オリジナリティ規範と自己検査 — 納品前の最終関門

この検査はオリジナリティ担保 2 層の**第 2 層 (制作エージェントの自己検査)**。第 1 層 = エージェント定義の絶対規則 (執筆中つねに有効)、第 2 層 = この自己検査 + マネージャー側の機械検証 (validate_song.py)。**外部 Web 照合は v0.5.0 で廃止**したので、自己検査が最終防波堤になる。役割は「不安を隠さず可視化し、**自分で書き直すか、確信が持てなければ P に握らせる**」こと。**検査を通すことではなく、不安を可視化することが目的** — 迷ったら申告に書く。

## 1. 絶対規則 (第 1 層の再掲。違反は検査以前の問題)

- 実在曲を参考にしてよいが、**歌詞原文・特徴的なメロディラインの直接流用は一切禁止** (引用・翻案・穴埋め改変・「あの曲っぽい言い回し」を思い出しながら書くことを含む)。ブリーフのモチーフから組み立てる
- 実在のアーティスト名・曲名・バンド名を歌詞・タイトルに書かない (言及は自己申告の出典欄のみ)
- 有名曲のサビ 1 行レベルの一致・タイトルの完全一致を避ける (一般語の部分一致は許容)
- 影響を受けた「雰囲気」の言語化は OK。それを P に隠さない (自己申告文化)

## 2. 検査手順 (3 観点)

### A. フレーズ検査 (行単位)

1. **検査対象行を抽出する**: サビ頭行 / サビ末行 / タイトルを含む行 / 決め台詞 (Bridge の告白行など) / 自分で「強い」と感じる行 — 計 4〜8 行
2. 各行について「**この行はどこから来たか**」を 1 行で答える。ブリーフのモチーフからどの展開 (04 §2) を経てこうなった、と説明できれば健全
3. **記憶由来を疑う兆候**が 1 つでもあれば、その行は要対処:
   - 書いた瞬間に、自分が設計した譜割りとは別のメロディ付きで浮かんだ
   - 特定の既存曲・アーティストの顔が浮かぶ
   - 「どこかで聞いた気がする」のに出所を説明できない
   - 慣用句を超えた長さ (7 モーラ以上 / 英語 4 語以上) の言い回しが一発で出てきて、修正の跡がない
4. 対処: §4 の書き直しプロトコルを実行するか、確信が持てなければ**自己申告の「不安のある行」に載せる** (自分で書き直すか、P に握らせる判断材料にする。Web 照合は廃止されたので、原則はこの場で書き直す)

補足: 頻出の一般表現 (「君に会いたい」「夜を越えて」級) は一致しても権利上の問題にならない。ただし一般表現だけで組まれた行は摩耗 (04 §6) として品質側で直す。

### B. 構造検査 (曲単位)

1. 構成表・感情曲線・モチーフ展開 (提示 / 反転 / 回収の位置) を並べて眺める
2. 「この並びは特定の 1 曲をそのままなぞっていないか」を自問する。**Verse–Chorus 型や V 字曲線のような定型・類型は共有財産なので問題ない**。問題になるのは特定曲固有の意匠の一致 (例: 特徴的な語りの入り方 + 同じ位置の転調 + 同じモチーフの回収、が全部揃う)
3. 特定曲が想起されたら: 構成要素を 1 つ以上ずらす (Bridge の内容を変える、回収位置を変える)。ずらさずに残す判断をしたなら「雰囲気の影響」として自己申告に書く

### C. タイトル検査

1. タイトルが**有名曲と完全一致していないか**を記憶ベースで確認する
2. 一般語 1〜2 語のタイトル (「サクラ」「ハナビ」級) は同名曲が多数存在しうる。完全一致が直ちに問題になるわけではないが、検索で埋没する実害があるため、限定を足した差別化案を併記する (「サクラ」→「終電のサクラ」)
3. 確信が持てない場合はタイトルも「不安のある行」に含めて申告し、差別化案を併記する (Web 照合は無いので、完全一致は自分で避ける)

### 補助検査: 自分の過去曲との重複 (アーティスト文脈のみ)

継続アーティストでは、他人の曲だけでなく**自作の使い回し**も見る (同じ決め行が 2 曲にあると、ディスコグラフィー全体の資産価値が下がる)。

- ブリーフに直近曲の要約や過去曲のフック行が含まれていれば、今回のサビ頭・サビ末との一致を確認する
- 情報が渡されていなければ検査不能なので、自己申告に「過去曲との重複は未検査 (情報なし)」と 1 行残す (渡されていないファイルを推測で探さない)

## 3. オリジナリティ自己申告 (納品時に必ず添える)

納品フォーマットの末尾に、次の 3 項目を埋めて添える。

```text
【オリジナリティ自己申告】
- 中心モチーフの出所: (ブリーフのどの要素から、どの展開パターンでこの表現に至ったか。1〜3 行)
- 意図的に避けた既存曲: (執筆中に接近を自覚して外した曲・言い回し。なければ「なし」)
- 不安のある行: (検査 A / C で確信が持てなかった行の全文を列挙。原則はこの場で書き直し、書き直せない/確信が持てないものだけ残して P に握らせる。なければ「なし」)
```

記入基準:

- 「なし」と書けるのは**検査を実施した上で該当がなかった**ときだけ (検査を省略しての「なし」は禁止)
- 「不安のある行」の閾値は低くする。**事故コストは信頼問題として高い**一方、外部照合は無いので、迷う行は自分で書き直すのが原則 — 書き直せないものだけ申告に残す
- 「意図的に避けた既存曲」には実在の曲名を挙げてよい (歌詞・タイトルへの使用ではなく、P への申告のため)。この欄は品質情報でもある (マネージャーが P に「この曲とは意図的に離してあります」と説明できる)

記入例 (粒度の目安。自作の想定曲):

```text
【オリジナリティ自己申告】
- 中心モチーフの出所: ブリーフの「夜の街 / 距離」から連想マップで「街の灯り」を採用。展開は反転型 (灯り = 存在の印 → 消えた窓 = 不在の印)
- 意図的に避けた既存曲: なし (執筆中に特定曲の想起はなかった)
- 不安のある行: 「まちのあかり ほどけて」— 灯りと「ほどける」の組み合わせをどこかで聞いた気がするが、出所を特定できない。§4 の書き直しプロトコルで別の具体物に振り直した
```

## 4. 書き直しプロトコル (検査 NG 時)

対象行が既存曲に近いと感じたときの手順。

1. その行が担っていた**機能**を構成表で確認する (フック / 傾斜 / 回収など)
2. 表現をいったん捨て、**ブリーフのコアモチーフに戻って別の具体物・別の動作で言い直す** (04 §6 の感情語翻訳と同じ要領)。元の行の語を 1〜2 語だけ差し替える「ずらし」は翻案になりうるため不可
3. モーラ数 (英語はストレス数) と韻計画は元の行に合わせて維持する (譜割りと構成を壊さない)
4. 書き直した行を検査 A にかけ直す
5. 修正が作品版に入ったら、入稿版を再変換する (06 の三原則)

## 5. 検査の出口条件

- [ ] A: 検査対象行 (4〜8 行) のすべてに「出所」の説明が付いた
- [ ] A: 記憶由来の兆候がある行は、書き直したか申告に載せた
- [ ] B: 特定曲の丸なぞりがない (定型の使用は問題ない)
- [ ] C: タイトルの完全一致確認を行い、必要なら差別化案を併記した
- [ ] アーティスト文脈では過去曲との重複を確認した (情報がなければ「未検査」と申告した)
- [ ] 自己申告 3 項目が「検査を実施した上で」埋まっている

# 11 タイトル設計 — 曲名の付け方

曲名は「聴く前に唯一見える言葉」であり、検索・共有・第一印象のすべてを背負う。この資料はタイトルの型・表記・造語・ダサさ回避・命名工程をまとめる。サビ頭/末への配置メカニクスとフック行からの切り出し (そのまま型/限定型/反転型) は 02 §2 が主管で、ここは「どんな型を、どの位置から、どう採り/採らず、どう審査するか」の総合資料。02 が配置、11 がタイトルの型と審査という分担にする。

## 0. この資料の約束

- 本文中の例タイトルはすべて自作、または一般類型としての記述であり、特定の実在曲を指さない。
- 実在曲のタイトルは §8 研究出典で「類型の実例」としてのみ言及する (タイトルは事実情報。歌詞原文は一切転載しない)。
- トレンドは古びる。命名の最新動向は suno-spec ではなくこの資料の範囲だが、鮮度は `:trend` 系リサーチで補完してから使う。

## 1. 大前提 — 「歌詞から採る=ダサい」ではない

調査の結論を最初に置く。歌詞 (特にサビ) からの抽出はプロの主流であり、洋楽ではフック=タイトルがデフォルト推奨ですらある。だから「歌詞の単語を使う」こと自体はダサさの原因ではない。

ダサくなるのは取り方の精度が低いときで、正体は次の 4 つ。

1. 機械的な切り出し (連呼語をそのまま採る、サビ頭を反射的に採る)
2. 説明過多 (テーマを直言してしまう)
3. 使い回しクリシェ (翼・奇跡・絆のような手垢のついた語の安易な結合)
4. 内容との不一致 (タイトル詐欺)

したがって本資料の方針は「歌詞流用の禁止」ではない。教えるのは、(a) どこをどう採るか (§2)、(b) 採らない選択肢=歌詞に出ない総括型を常に手札に持つこと (§2 の④)、(c) 型・表記・造語・審査という道具立て (§3〜§7) である。

発案者の直感との照合 (忖度せず正直に): 「歌詞の単語をそのまま使うパターンが多すぎる」という指摘は**部分的に反証された** — 問題は流用そのものではなく取り方にある。一方で「よほどキャッチーで語感の良いフレーズが歌詞にあればそのまま採る」「造語は 1 オプションで乱用しない」は調査でも支持された。

## 2. タイトルと歌詞の関係 (抽出位置)

タイトルをどこから採るかには 4 つの位置がある。近年の傾向まで込みで手札にする。

| 位置 | 何を採るか | 近年傾向 | 実務メモ |
|---|---|---|---|
| ① サビのリフレイン | 連呼される語をそのまま | 減少 | 最も付けやすいが機械的になりやすい。②③と比べてから決める |
| ② サビ終わりのキメ | サビ末の決め言葉 | 最多・推奨 | 現在の第一候補。サビの「残す 1 行」(02 §2) から採る |
| ③ 歌詞内キーワード | サビ以外の印象語 | 中間 | Verse/Bridge で一度だけ光る具体語を拾う |
| ④ 総括型 (歌詞に出ない) | 歌詞に出ない言葉で本質を言い当てる | 増加中 | 常に選択肢に入れる。説明ではなく「本質の一語」 |

- キャッチーで語感の良いフレーズが歌詞にあれば、そのまま採るのは強く有効である (検索での発見可能性が最大化する)。無理に捻らない。
- ④ の総括型 (歌詞に出ないタイトル) について、02 §2 はタイトル=フック一致を原則とし歌詞に出ない設計を「あえての判断」とするが、本資料はそれを常設の第 4 の選択肢として扱う。矛盾ではなく分担で、02 が配置の原則、11 が「採らない選択肢」の常備を担う。①②が凡庸なときは、④で本質を突けないか必ず一度試す。
- 配置 (サビ頭か末か) と反復回数は 02 §2 / §3、タイトルの完全一致検査は 08 §2-C。ここでは重複させず接続する。

## 3. 類型カタログ

12 の型。排他ではなく組み合わせ可能。例はすべて自作。

| 類型 | 効く理由 (1 行) | 使いどころ | 再現手順 (簡潔) |
|---|---|---|---|
| 一語・体言止め | 記憶・検索・言及が最速。ストリーミング期に構造的有利 | ジャンル問わず第一候補 | 感情/情景の名詞を 20 個出す → 口に出して締まる語を残す → 被り検索 |
| 漢字圧縮 (一〜四字) | 少ない字数に多義を詰め、硬質・文学的に見える | 激情・和・アニメ世界観 | 和語/古語/仏教語を探索 → 無ければ二字を新結合 → 読みが立つか確認 |
| ひらがな開き | 優しさ・素朴・儚さを表記で運ぶ | 内省・喪失・柔らかい曲 | 候補を漢字/カナ/かなで並べ書き → 感情温度に合う表記を選ぶ |
| 英語・カタカナ一語 | 検索性とジャンル感の即時シグナル | 洋楽感・K-POP 感・グローバル狙い | テーマを英訳 → 一語名詞化 → カナ音写も試し被りで選ぶ |
| 口語フレーズ | 当事者性が出てミーム化しやすい | SNS 拡散前提・キャラ立て | 主人公が実際に口に出す台詞を書く → 5〜10 字に刈る → 語尾で温度調整 |
| 文章型 (名詞+助詞+動詞) | 認識しやすさと独自性を両立し、物語が起動する | バラード・物語性の曲 | 核の名詞+動詞を選ぶ → 助詞を差し替えて語感比較 → 被り検索 |
| 長文型 | 一覧で物理的に目立ち、題だけで物語が完結する | 差別化の飛び道具 (乱発しない) | サビ核文/心内語を一文に → UI で切れても前半で意味が立つ語順に |
| 造語・当て字 | 検索独占。定着すれば強烈に残る (§5 が主管) | 唯一性が要るとき限定 | §5 の手順。審査を通ったときだけ採用 |
| 記号・スタイリング | 画面上で音量・テンションを伝える唯一の手段 (§4 が主管) | ダイナミクスを表記で示したいとき | §4。大小文字/記号ありなしを並べて選ぶ |
| 二重読み | 聴く前と後で意味が反転しリピートの発見を生む | 反転構造・伏線のある曲 | 候補にポジ読み/ネガ読みを書き出し両立語だけ残す |
| 異質結合 | 「どういう意味?」の謎がクリックを誘発する | フックの弱いテーマの底上げ | テーマ語を固定 → 遠い名詞を総当たり → 意味が発生する組だけ残す |
| 象徴物 (斜めに言う) | 説明を避け聴後の「そういうことか」を作る | テーマ直言を避けたいとき | 直言候補を全部捨てる → 歌詞世界に置ける具体物を列挙 → 遠いのに繋がる物を選ぶ (花言葉の逆引きも可) |

自作例: 一語「夜光」/ 漢字圧縮「灯芯」/ ひらがな「ほどける」/ カタカナ「アンダートウ」/ 口語「もう帰ろっか」/ 文章「星をこぼす」/ 長文「終電を見送って僕らはまだ何者でもない」/ 異質結合「制服とロケット」/ 象徴物 (別れを霜柱に託す)「霜柱」。

## 4. 表記の設計 (独立項目)

同じ音でも表記が変われば運ぶ感情が変わる。表記はメロディやモチーフと同格の、独立した設計項目として立てる。

| 表記 | 運ぶ感情・印象 |
|---|---|
| ひらがな | 優しさ・素朴・儚さ・脆さ。子供性・体温 |
| カタカナ | クール・無機質・記号性。日本語文脈では準造語化して被りが減る |
| 漢字 | 硬質・文学的・様式。字面で世界観を立てる |
| 英字 | スマート・グローバル・ジャンル感。小文字=内省、全大文字=激情の目安 |
| 記号 (! 。 、 …) | 音のない画面で音量・余韻・勢いを刻む |

- 決定稿は必ず 3〜4 表記で並べ書きし、曲の感情温度に合う 1 つを選ぶ。
- **アーティスト名と横並びで見る**。プレイリストやジャケットでは常に「アーティスト名 — 曲名」の対で表示される。名前の表記 (artist.yaml) と衝突・重複しないか、対で見て締まるかまで確認する。
- 日本語曲でも英字・カタカナ題は有効だが、意味の裏付けなしに「スマートに見せたいだけ」で変換すると空回りする (§6)。
- 表記いじりの最終形は入稿版の変換規則 (06 の R 群) とは別物である。作品版タイトルの表記が正で、入稿時の発音対策とは分けて考える。

## 5. 造語の扱い (1 オプション、乱用しない)

造語には強さと弱さの両面がある。既定で造語に寄せない。

グラデーション (どれも「ゼロからの音の羅列」より、意味の乗った半造語が定石):

1. 音写準造語 (外来語の音をカナで写す。表記で唯一性を得る)
2. 逆転・再結合 (既存の複合語を組み替える)
3. 架空固有名 (実在しない地名・人名)
4. 漢字新結合 (実在熟語がなければ二字を組む)

- 強み: 検索を独占できる。定着すれば強烈に記憶される。
- 弱み: 認知に時間がかかる。読めない語・複数読みの語 (アイ/エーアイ問題) は検索も口コミも割る。
- **結論**: 造語は 1 つの強力なオプションであって全体戦略ではない。曲ごとに既定で造語へ寄せず、§7 の審査を通ったときだけ採用する。造語だらけの作家性はこのプロダクションでは狙わない (発案者の要望と調査が一致した点)。
- 例外: 難読・造語が「引き」になるニッチ (ボカロの熟語美学など) は、ジャンルの読解リテラシーに依存する。ターゲット層を確認してから使う。

## 6. ダサさの禁止パターン表

| 禁止パターン | 症状 | 代わりにどうするか |
|---|---|---|
| テーマ直言・説明過多 | 「愛」「別れ」等をそのまま置く、状況を説明した題 | 象徴物型 (§3) で斜めに言う。20〜30% の謎を残す |
| 使い回しクリシェ | 翼・奇跡・絆・涙の手垢のついた結合 | 具体物・固有イメージに置換。異質結合 (§3) で更新する |
| 時代感のズレ | ネットスラング・流行語で経年劣化する | 語尾/スラングは風化前提。普遍語に核を置き、流行は薬味に留める |
| 裏付けのない表記いじり | 意味もなく安易にカタカナ化・英字化する | 表記が運ぶ感情 (§4) と曲の温度が一致するときだけ変える |
| 内容との不一致 (タイトル詐欺) | 題と歌詞世界がずれている | §7 の内容整合審査で落とす。歌詞完成後に再照合する |

留保: ダサさは強く主観的であり、「型として正しい」ことと「万人に刺さる」ことは別である。審査は候補を落とすためではなく、外す理由を言語化するために使う。

## 7. 命名ワークフロー

天啓ではなく工程で作る。

1. **多案出し**: 10〜20 案を目安に、複数の類型 (§3) をまたいで出す。全部を同じ型・同じ位置 (サビ頭) に寄せない。②サビ末・③印象語・④総括型を必ず混ぜる。
2. **客観審査**: 各案を次で絞る。
   - 見た目: アーティスト名と横並びにする (§4)
   - 響き: 声に出す。歌ものとして締まるか
   - 表記比較: 漢字/かな/カナ/英字/記号を並べる (§4)
   - 検索被り: 配信・動画で候補名を検索する。強い既存曲と被れば語形・表記を変える (完全一致検査は 08 §2-C)
   - 内容整合: 歌詞世界と一致するか (タイトル詐欺の排除)
3. **3 秒の第一印象**: 聴き手は数秒で判断する。前提知識のない目で見て「気になる」か。
4. **仮タイトル運用**: 仮題で書き始め、歌詞完成後に §2 の位置と §3 の型で再審査してよい (詞を書き切ってから最後に付ける後付け型も実務的)。
- 納品では、審査を通った**本命 1 + 対抗 1〜2** に絞って提示する (作詞家の【タイトル案】)。
- **造語・凝った型 (二重読み・異質結合・長文) は、この審査を通ったときだけ採用する**。通らなければ一語・体言止めや文章型の平明な案に戻す。
