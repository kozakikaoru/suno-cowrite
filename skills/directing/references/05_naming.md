# 05 ネーミング — 固有名資産の設計 (上位ネーミング)

アーティスト名・シリーズ/連作名・EP/アルバム名・世界観固有語を決めるときに読む。**曲名はこの資料の対象外で、`skills/songwriting/references/11_title-craft.md` (lyricist 主管) が正**。この資料が扱うのは曲より上位の「一度決めたら長期に固定される固有名資産」。曲名は毎回変わるがアーティスト名は変えられない、という要件の違いが分界の理由。命名の道具立て (類型カタログ・表記設計・造語グラデーション・審査ワークフロー) は 11 が持つ資産であり、**ここでは再導出せず参照する**。05 が足すのは上位ネーミング固有の関心 (長期運用性・全曲横断の一貫性・用語体系の整合) だけ。

> - 例 (名前・造語・用語) はすべてこの資料のための自作。実在アーティスト・ブランド・レーベル名は本文に書かない。
> - 命名の型・表記・造語・審査の道具立ては songwriting 11_title-craft が主管。この資料はそれを再掲せず参照する (重複させない)。
> - 判定材料は artist.yaml / world.md / discography.md が正。この資料は「どう設計・審査するか」の方法論だけを持つ。
> - 実務値 (商標・配信ハンドル・ドメインの空き) は取得時点で変わる。最新は都度確認し、命名トレンドは `:trend` で補う。

## 1. ネーミングの 5 層と主管

- 曲名は「1 曲の産物」で毎回変わり、完成した歌詞から抽出・確定される (11 §2) ので **lyricist** が主管。アーティスト名・シリーズ名・世界観語は「曲の上位にあり全ディスコグラフィーで一貫させる資産」で長期固定なので **director** が主管。altitude ロジック (上位契約 vs 実装) は命名でも一貫する。

| 層 | 主管 | 置き場 |
|---|---|---|
| 曲名 | **lyricist** (11_title-craft) | song.json の title・notes |
| EP / アルバム名 | **director** (§3) | discography・direction.md |
| シリーズ・連作名 | **director** (§3) | discography・world.md |
| アーティスト名 | **director** (§2) | artist.yaml の name / name_en |
| 世界観固有語 (造語世界) | **director** (§4) | world.md 語彙パレット |

- 曲名を付ける・命名の型/表記/造語/審査の道具立てを使うときは 11 を読む。director の既定配線に 11 のパスは常時入れない (クロスライブラリの常時結合を作らない)。深掘りが要るとき (アーティスト名を本気で詰める debut 等) だけ、マネージャーが 11 のパスを添える。

## 2. アーティスト名の設計 — 第一要件は「空き確認の工程」

- アーティスト名は変えられない (曲名は毎回変わる)。ゆえに語感の審査より先に「実務チェック」が来る。ここが曲名との最大の差分。
- 第一要件 = 空き確認の工程 (機械的に絞る):
  1. 候補を複数出す (語感審査はこの後)。
  2. 空き確認先で機械的に絞る: Google 検索 / 各配信 (発見可能性) / SNS ハンドル / ドメイン / 商標データベース。
  3. **同名の別アクトと衝突しないこと**を最優先条件にする。
- 一意性の作り方: 他と語を共有するなら語を足す、または綴りを意図的に変える (既存語のスペル替え)。一意性の確保と商標回避を同時に得られる。
- 記憶性: 読みやすい = ロゴ化しやすい (04 §1 のロゴ 40px 要件と接続)、言いやすい = 口コミで広がる。
- 表記戦略 (artist.yaml の name / name_en): 表記が運ぶ感情 (ひらがな/カタカナ/漢字/英字) は 11 §4 が主管。ここで足す上位固有の軸は次の 3 点。
  - **長期固定に耐えるか** — 流行語・時事ネタ・年号を含めない (曲名より風化耐性を重視する)。
  - **海外読み・誤読の許容範囲** — グローバル配信での発音・検索を想定する。
  - **ロゴ/ワードマーク化した時の形** — 04 §1 のロゴ要件 (40px・単色) と対で見る。
- ネット発の三層命名: 制作主体名 (あなた) / 表に立つアーティスト・ユニット名 / 作品群の傘 = シリーズ・レーベル名 を、別レイヤーの固有名資産として分けて設計できる。聴衆が贈る・あるいは自ら名乗る「称号サフィックス」の文化もこの系譜 (本プラグインの呼称文化の背景)。
- 命名の道具立て (類型カタログ・造語グラデーション) は 11 §3・§5 を参照する。ここで再導出しない。

## 3. シリーズ・連作 / EP・アルバム名

- まとめ性: 複数曲を一つの傘でくくる名。ディスコグラフィーの物語 (03) と接続させ、「この一群は何の章か」を名前で示す。
- ナンバリングとサブタイトル: 連作でナラティブを張る (巻数・章立て + サブタイトル)。サブタイトルは era (04 §5 の可変層) と対応させると、視覚の変化と名前の変化が揃う。
- 一貫性: シリーズ名は世界観語彙 (§4) と同じ命名言語に属させ、world.md 語彙パレットと整合させる。傘の名前だけ世界観から浮くと束ね効果が弱る。
- EP/アルバム名の審査は §5。曲名の型・表記は 11 が主管で、ここは「まとめ性・横断一貫性」の観点だけを足す。

## 4. 世界観固有語・造語体系 — 「命名言語」として設計する

- 造語を単発で作らない。使う音・文字のサブセット (**命名言語** naming language) を先に決め、それだけで作る。これで固有名群が「同じ言語に属する」ように見え、世界が一貫して実在感を持つ。
- 作り方の型: 一から作らず古い語 (ラテン/ギリシャ/古語) から借りる、または既存語を再目的化する (合成語にすると造語なのに覚えやすい)。**中核語を数個に絞り、周辺は既存語の再目的化で薄める** (造語の認知コストは高いため)。
- 乱用を避ける規律 (最重要): **控えめに投下し、キーフレーズは反復して覚えさせる**。不用意な大量投下は聴衆を圧倒して離脱させる。少量ずつ撒き、繰り返して定着させる。
- 識別性: 固有名同士を取り違えないように作る。同じ頭文字 × 同じ種類、似た長さ + 似た母音の語は (特に音声で) 混同を招く。アーティスト名/シリーズ名の一意性 (§2) と同じ原理。
- 非西洋の実在語を「異世界語」として流用しない (不自然・盗用的に映る)。本プラグインの「世界観はオリジナル」原則と一致する。
- 読みの固定: world.md 語彙パレットに読み (かな) を併記して固定する。造語の型・当て字の手順は 11 §5 を参照する。

## 5. 上位ネーミングの審査

- 11 §7 の命名ワークフロー (多案出し → 客観審査 → 3 秒の第一印象) を上位ネーミングにも転用する。ワークフロー本体は再掲せず、上位固有の追加軸だけを足す。
- 追加軸 (上位ネーミング固有):
  - **長期性** — 5 年後も古びないか (流行語・時事・年号を避ける)。曲名より風化耐性を重視する。
  - **横断一貫性** — 全曲・全 era で一貫する資産として整合するか。アーティスト名 ⇄ シリーズ名 ⇄ 世界観語が同じ命名言語 (§4) に属するか。
  - **用語体系の整合** — §4 の命名言語のサブセットから外れていないか。
  - **空き確認** — アーティスト名は必須 (§2)。シリーズ名も強い既存作と被らないか確認する。
- 曲名の審査 (表記比較・検索被り・内容整合・完全一致検査) は 11 §7 と 08 §2-C が主管。ここは重複させず、上位資産の「長期・横断」だけを見る。
