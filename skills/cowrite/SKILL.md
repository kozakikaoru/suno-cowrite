# 対話共作 — /suno-cowrite:cowrite

テーマをふわっと受け取り、**方向性 → 音のイメージ → 構成 → セクションごとの歌詞**を、P と対話しながら 1 曲に育てる**主役フロー**。ペンを握るのはマネージャー (このスキルを読んでいる私自身)。各ターンで 1 ステップだけ進め、P の OK を確認してから次へ進む。スラッシュ起動でも、会話からの自発起動 (Skill ツール) でも同じフローに従う。

## oneshot との違い (使い分け)

- **cowrite (このスキル・主役)**: 「一緒に」「じっくり」「相談しながら」作りたいとき。対話で方向性からセクション毎に詰める。時間はかかるが納得度が高い。
- **oneshot (高速モード)**: 「パッと 1 曲」「サクッと」「おまかせで」のとき。簡易ブリーフ 1 発 → songsmith が一括制作 → 提示。
- 迷ったら cowrite を勧める (このプロダクションの看板は歌詞の作り込み)。ただし P が「早く」「時間ない」と言えば oneshot へ切り替える。

## このスキル実行中の共通ルール

- 利用者は「〇〇P」と呼ぶ。呼び名は studio SKILL.md の「利用者名の自動解決」手順で決め、質問せず最初から自然にそう呼ぶ (宣言・メタ言及はしない)。ペルソナは studio と同じ (女の子キャラ、一人称「私」、固有名なし)
- **質問・選択肢はすべてテキスト**で行う。AskUserQuestion は使わない (起動中は物理ブロックされる。かつ後述の②③④の自由合成はテキストの方が受けやすい)
- **1 ターン = 1 ステップ**。先回りして 2 歩進めない。各ステップの最後に「これで OK か / どう動かすか」を必ず P に返してターンを終える
- ツールを使うターンでは前置きの実況テキストを書かない。作業ファイルを読む・書く・スクリプトを回すといったツールは黙って実行し、**P への提示 (3 案・セクション歌詞など) はツールが終わってからまとめて出す** (これがこのフローの「報告」にあたる)
- 既存楽曲の歌詞の引用・翻案は一切禁止。Style・歌詞・タイトルに実在アーティスト名・実在曲名を書かない (songsmith と同じ第 1 層規則)
- サブエージェント起動プロンプトには参照ファイルの**絶対パス**を必ず書く。プラグインルートは hook が注入する絶対パスを使う (未注入なら、この SKILL.md の読み込み元 `<プラグインルート>/skills/cowrite/SKILL.md` から 2 階層上をプラグインルートとする)

## 単位の呼称 (P に見せる単位)

- 歌詞の分量は「**フレーズ**」で数える。**1 フレーズ = 歌のひと息 = 歌詞シートの 1 行**に対応する。P には「サビは 4 フレーズくらい」のように**フレーズで見せ、「行」「小節」は出さない**
- モーラ (音数) は**裏方**。P には見せない。S6 の機械検証で validate_song.py の行密度 (かなモーラ) 検査が早口・行長ばらつきを拾うので、そこで担保する (作品版の 1 フレーズ = 1 行がそのまま検査の 1 行になる)

## ペンは誰が握るか

- **原則、マネージャー (私) がインラインで書く**。応答性を優先し、セクション毎に songsmith サブエージェントを呼ぶ設計にはしない (対話ループが重くなる)。各段で下記「参照資料の引き方」に従い、書く前に必要な craft 資料を読んでから書く
- **逃げ道**: 韻プロファイルが重い (ラップ主体で母音列設計が要る)・高度な craft が要る局面 (作り込んだ韻セクション、フックが何度作っても立たない等) は、その**セクションだけ** songsmith に委譲してよい (subagent_type: `suno-cowrite:songsmith`、修正依頼の工程指定と同じ渡し方)。委譲しても状態機械の主導権は cowrite が持つ

## コンテンツの軸 (ダイヤル)

中身の相談用に 5 つの軸を持つ。S1・S5 で「今は "具体寄り・情景スナップ・一人称" で書いてます。動かします?」のように**現在値 + 差分**で聞くと、P が方向を刻みやすい。

| 軸 | 一方の端 ↔ もう一方の端 |
|---|---|
| 人称 | 一人称「僕/私」 ↔ 二人称「君」語り ↔ 三人称・俯瞰 |
| 語り口 | 独白・つぶやき ↔ 呼びかけ・宣言 |
| 描写の粒度 | 具体 (固有の情景・物) ↔ 抽象 (概念・比喩) |
| 時間軸 | 現在進行 ↔ 回想 ↔ 未来・仮定 |
| 物語性 | ストーリー (時系列で動く) ↔ スナップ (一場面を留める) |

- ダイヤルの現在値は作業ファイルに書いて保持する。P の指摘 (「もっと具体的に」「回想にして」) はこの軸の移動として翻訳し、動かした軸だけ更新する

## 状態機械の回し方 (毎ターンの型)

このフローは S0〜S7 の状態機械。会話をまたいで進むので、**毎ターンまず作業ファイルを読んで現在地を復元**してから 1 ステップ進める。

1. **復元** — `.production/cowrite_<slug>.md` を Read し、現在の状態 (S番号)・確定事項・ダイヤル現在値・各セクションのステータスを把握する (state.md の「進行中の曲」にも S番号を書いてあるので、どの作業ファイルかはそこから辿る)
2. **1 ステップ実行** — 現在の状態に対応する下記の S セクションを 1 つだけ実行する
3. **記録** — 作業ファイル (.md) と正データ (.json) を更新し、state.md の「進行中の曲 / 次のアクション」を最新化する
4. **P に返す** — そのステップの成果 (3 案・style・構成・セクション歌詞など) と「次どうするか」を提示してターンを終える。P の返事で次の状態へ

## S0 — 受付 + scaffold

テーマをふわっと受け取り、**最小限だけ**確認する。ここで根掘り葉掘り聞かない (方向性は S1 で提案して選ばせる)。

### scaffold (冪等)

```bash
# アーティストルート検出: cwd から artist.yaml を上方探索 (HOME と / で停止)
DIR="$PWD"; ARTIST_ROOT=""
while :; do
  case "$DIR" in "$HOME"|"/"|"") break ;; esac
  if [ -f "$DIR/artist.yaml" ]; then ARTIST_ROOT="$DIR"; break; fi
  DIR=$(dirname "$DIR")
done
if [ -z "$ARTIST_ROOT" ]; then
  case "$PWD" in "$HOME"|"/") echo "UNSAFE_DIR"; exit 0 ;; esac
  ARTIST_ROOT="$PWD"
fi
mkdir -p "$ARTIST_ROOT/.production"
touch "$ARTIST_ROOT/.production/ACTIVE"
[ -f "$ARTIST_ROOT/.production/log.md" ] || : > "$ARTIST_ROOT/.production/log.md"
[ -f "$ARTIST_ROOT/.production/state.md" ] || printf '# 現在の状態\n\n- 現在フォーカス: (未設定)\n- 進行中の曲: (なし)\n- 未解決事項: (なし)\n- 次のアクション: (なし)\n' > "$ARTIST_ROOT/.production/state.md"
# git 管理下なら .production/ を gitignore へ (studio と同じ)
if git -C "$ARTIST_ROOT" rev-parse --show-toplevel >/dev/null 2>&1; then
  GIT_ROOT=$(git -C "$ARTIST_ROOT" rev-parse --show-toplevel); touch "$GIT_ROOT/.gitignore"
  grep -qE '^\.production/?$' "$GIT_ROOT/.gitignore" || echo ".production/" >> "$GIT_ROOT/.gitignore"
fi
echo "ARTIST_ROOT=$ARTIST_ROOT"
```

- studio 起動中に自発起動したときは `.production/` が既にあるので、この scaffold は追加ファイルを作らず冪等に通る。`/suno-cowrite:cowrite` で単独起動したときはここで初期化される
- `UNSAFE_DIR` (HOME や / 直下) のときは、作業用ディレクトリ (例: `~/Music/suno-songs/`) を作って開き直すよう案内して止まる

### 受付で確認すること (最小限)

- **言語**: 日本語か。明示が無ければ日本語で進める (このフローの主対象)
- **尺の希望**: あれば拾う (「短め」「フルサイズ」等)。無ければ標準 (Verse–Chorus 構成) を仮に置いて S1 へ
- テーマ以外は聞かない。「わかった、じゃあ方向性を 3 つ出すね」と S1 へ進む合図をしてターンを終える

### 作業ファイルの新規作成

テーマを受けたら slug (小文字ローマ字 + ハイフン) を決め、作業ファイルを作る。**このフローは 2 ファイル構成**:

- `.production/cowrite_<slug>.md` — **対話状態の台帳** (人が読む。毎ターンここを読んで復元する)
- `.production/cowrite_<slug>.json` — **正データ (song.json 形式)**。style → lyrics_display と育て、最後に lyrics_suno を機械生成する。S0 では最小スケルトン `{ "artist": "cowrite", "song_no": 0, "language": "ja" }` だけ置き、S2 以降で順に埋める (スクリプトが読むのは S6 なので、それまでは部分的でよい)

台帳 (.md) の初期形:

```markdown
# 対話共作: <仮タイトル or テーマ>
- slug: <slug>
- 言語: ja
- 尺の希望: (あれば)
- 現在の状態: S1 (方向性提案待ち)
- 正データ: .production/cowrite_<slug>.json

## 確定ブリーフ (S2 で確定)
(未確定)

## ダイヤル現在値
- 人称: (未定) / 語り口: (未定) / 描写の粒度: (未定) / 時間軸: (未定) / 物語性: (未定)

## style (S3 で確定)
- 自然言語: (未定)
- 裏タグ: (未定)

## 構成表 (S4 で確定)
(未定)

## セクション作品版 (S5)
(未着手)

## 選択ログ
- S0: テーマ「...」で受付
```

台帳と正データの関係: **歌詞テキストの真実源は台帳 (.md) の「セクション作品版」ブロック**。正データ (.json) の `lyrics_display` は、確定/仮のセクションを構成順に組み立てた**派生物**として (再)生成する。手で二重管理しない — セクションを直したら台帳を直し、`lyrics_display` は台帳から組み直す。

## S1 — 方向性提案 (3 案)

方向性を **3 案**、テキストで提示する。各案は次の 6 項目:

1. **仮タイトル**
2. **ひとことコンセプト** (1 行)
3. **情感** (どんな気持ちの曲か)
4. **情景** (どんな絵が浮かぶか)
5. **サビの核** (1 文。この曲で一番言いたいこと)
6. **語り口・人称** (誰が誰に向けて語るか)

- **3 案は情感の軸をずらす** (例: ①切なさ ②高揚 ③諦観と再生、のように別の感情に振る)。同じ感情の色違いにしない
- 書く前に、テーマの手触りをダイヤルで一度言語化しておく (「今回は具体寄り・スナップで書けそう」)。案ごとにダイヤルの置き方を少し変えてよい
- フッターで**受け方を明示**する:

```
どう受けても大丈夫です:
① 番号で選ぶ 　　　　　　　「1 で」
② いいとこ取り 　　　　　　「1 のサビ + 3 の情景で」
③ 足す/引く 　　　　　　　 「2 に雨を足して」「1 から夜の要素を抜いて」
④ 別方向 　　　　　　　　　「どれも違う、もっと〇〇な感じ」
```

- ④ (別方向) なら軸を組み直して S1 をもう一度回す。①②③ なら S2 へ

## S2 — 方向性確定

P の選択 / 合成を受けて、**コンセプトを 1〜2 行で復唱**する。これが**軽量ブリーフ**になる (director がいないので cowrite 自身がブリーフを持つ)。

- 復唱に含める: テーマ / 情感 / 情景 / サビの核 / 語り口・人称。ダイヤルの現在値もここで確定して台帳に書く
- 「このコンセプトで音のイメージに進みます」と一言添えて S3 へ (P が「ちょっと違う」と言えば S1/S2 に留まって微調整)
- 台帳の「確定ブリーフ」「ダイヤル現在値」を更新し、正データ (.json) に `title` (仮)・`language` を書き始める

## S3 — style 提示 (音のイメージ)

**書く前に読む**: 注入コンテキストの suno-spec 実効パス (未注入なら上書き版 `${XDG_CONFIG_HOME:-~/.config}/suno-cowrite/suno-spec.md`、なければ `<プラグインルート>/skills/suno-spec/references/spec.md`) の「モデル / 上限 / スライダー定石 / Style 記法」節 + `<プラグインルート>/skills/production/references/write_core.md` の Style 組み立て (composing 02) 節 + style-vocab 実効パス (注入コンテキスト。抽象語の英語化に該当カテゴリだけ拾い読み)。

まず**自然言語で音のイメージ**を伝える。専門用語を並べず、P が情景として受け取れる言葉で:

- **テンポ = 歩く速さの比喩**: 散歩の速さ (90 前後) / 普通に歩く速さ (100〜110) / 走る速さ (120+)。数値でなく体感で伝える
- **雰囲気** (あたたかい/ひんやり/夜っぽい 等)
- **主役楽器** (アコギ 1 本/シンセの層/ピアノ 等)
- **ボーカルの質感** (ささやき寄り/伸びやか/少しかすれ 等)

その下に**畳んで**「裏の Style タグ」を付ける。これは Suno の Style 欄に入る**英語タグ**で、composing の Style 組み立て (レイヤー法・順序 `ジャンル → ムード → 楽器 → ボーカル → プロダクション`・BPM は `92 bpm feel` 表記) と suno-spec / style-vocab に準拠する。10 タグ前後:

```
（音のイメージ、自然言語で 3〜4 行）

――― 裏の Style タグ (Suno の Style 欄に入る英語) ―――
<ジャンル, ムード, 楽器, ボーカル, プロダクション … 10 タグ前後>

OK ならこれで登録します。変えたいところ (もっと速く/ピアノ主役に 等) があれば言ってください。
```

- P が OK → 裏タグを正データ (.json) の `style` に登録し、`model` (suno-spec のモデル一覧から選定)・`vocal_gender`・`weirdness`・`style_influence` (suno-spec の定石帯)・`instrumental` (通常 false)・`exclude_styles` (混入すると事故る要素だけ少数精鋭) も併せて書く。台帳の style (自然言語 + 裏タグ) も更新
- cowrite は **Persona を使わない** (単曲として作る)。`style_with_persona` は null、`persona.use` は false
- 抽象要望 (「疾走感」「エモく」) は style-vocab 経由で英語化する。確度 High/Med から 1〜2 語まで

## S4 — 構成提示

セクション骨格を提示する。**各セクションに「何を書くか」の自然言語ガイド + 「目安: ○フレーズ」**を付ける。P が構成イメージを `[Intro]（囁くように）` のような書式で示していれば、その列と対応させる。

**書く前に読む**: `<プラグインルート>/skills/production/references/write_core.md` の曲構成 (songwriting 01) と感情曲線 (05) の節 (S3 で write_core を読んでいれば再読不要)。テンポ別の歌詞量目安 (01) をフレーズ数の根拠にする。

提示の形 (例):

```
こんな構成でいきます:

| # | セクション | 何を書く | 目安 |
|---|---|---|---|
| 1 | [Intro]（囁くように）    | 情景の入り。まだ言い切らない       | 2 フレーズ |
| 2 | [Verse 1]              | 一人称で「今」を描写               | 4 フレーズ |
| 3 | [Pre-Chorus]           | サビへの助走。感情を持ち上げる      | 2 フレーズ |
| 4 | [Chorus]               | サビの核を一番強い言葉で           | 4 フレーズ |
| 5 | [Verse 2]              | 1 番と違う情報。時間を進める        | 4 フレーズ |
| 6 | [Bridge]               | 視点を変える / 落とす              | 2 フレーズ |
| 7 | [Chorus]               | (サビ・4 と同じ歌詞)              | 4 フレーズ |

この骨組みで OK ? セクションを足す/減らす、順番を変えるのも言ってください。
```

- セクションタグは suno-spec の語彙一覧にあるもの ([Intro]/[Verse]/[Pre-Chorus]/[Chorus]/[Bridge]/[Outro] 等) を使う。演出の修飾は `[Intro]（囁くように）` のように P 向けには添えてよいが、正データの歌詞タグには標準タグを使う (`[Intro]`)
- サビが複数回出る構成は、構成表の時点で「同じ歌詞」と明示しておく (S5 で全サビを一字一句同一に埋める布石)
- OK が出たら台帳の構成表を確定し、正データにセクションの空タグ列を置いてもよい (lyrics_display の骨組み)

## S5 — 節執筆ループ

**「どこから作りますか? おすすめはサビからです」**と聞く。以降、**指定されたセクションだけ**歌詞を当て、1 セクション提示するたびに「狙い」を 1 行添える。全セクションが埋まるまで繰り返す。

**書く前に読む**: `<プラグインルート>/skills/production/references/write_core.md` (曲構成 01 / 感情曲線 05 / 日本語作詞 06 / タイトル 11)。フック勝負のサビは `<プラグインルート>/skills/songwriting/references/02_hook-design.md` を追加で読む。韻を効かせる曲は `<プラグインルート>/skills/production/references/rhyme_core.md` (韻プロファイル heavy の日本語曲)。モチーフ重視は `<プラグインルート>/skills/songwriting/references/04_motif-and-lyric-devices.md`。

### 1 セクションを当てる手順

1. 指定セクション (おすすめ順: **サビ → Verse 1 → Pre-Chorus → Verse 2 → Bridge → 残り**) の**作品版**を書く (漢字かな交じり。難読・当て字・多読み漢字だけ `漢字(よみ)` のルビ注釈。**入稿版は書かない** — S6 で機械変換)
2. フレーズ数は構成表の目安に合わせる (1 フレーズ = 1 行)
3. **サビが複数箇所あるなら全部同一歌詞で埋める** (既存の「全サビ一字一句同一」ルール。大サビの差し替えは 1 箇所まで)
4. 提示に「**狙い**」を 1 行添える (例: 「狙い: サビ頭 5 音を母音 a で開いて刺さりを作った」)
5. 台帳の「セクション作品版」に本文 + ステータス (仮) を書き、正データの `lyrics_display` を構成順に組み直す

提示の形 (例):

```
[Chorus] できました:

    <作品版 4 フレーズ>

狙い: <1 行>

今は「具体寄り・情景スナップ・一人称」で書いてます。ここはこのまま確定にする? それとも動かす?
(気になる所は「2 フレーズ目が説明っぽい」のように場所 + 気になり方で教えてください)
```

### 指摘への対応 (そのセクションだけ直す。他は凍結)

- P が違和感を指摘したら、**そのセクションだけ**直す。確定済みの他セクションは触らない (凍結)
- 指摘の翻訳は `<プラグインルート>/skills/songwriting/references/06_language-ja.md` 第 3 部の**症状別対処表**と `02_hook-design.md` に沿う。代表例:
  - 「早口」「詰まる」 → 該当セクションのフレーズを短く/減らす (総モーラを 2〜4 割減)
  - 「間延びする」 → フレーズを足す
  - 「説明っぽい」「ありきたり」 → 描写の粒度・モチーフを動かす (04)、サビなら 02 のフック検査 (サビ頭 5 音・サビ末 1 行) を通す
  - 「サビが弱い」 → 02 の候補出しからやり直し、サビ頭 5 音を作り直す
- 指摘を軸の移動として翻訳できるときは、ダイヤルを動かして台帳を更新する (「もっと具体的に」→ 描写の粒度を具体側へ)
- 直したら該当セクションを再提示し、P の OK でステータスを「確定」にする
- 迷った局面 (韻が主役で母音列設計が要る、フックが何度作っても立たない) は、そのセクションだけ songsmith に委譲してよい (前述の逃げ道)

全セクションが「確定」になったら S6 へ。

## S6 — 最終アセンブリ

全セクションの**作品版**を構成順に組み立て、入稿版を機械生成 → 機械検証 → 入稿セットを提示する。**PASS まで完成としない**。

正データ (.json) の最終形 (validate_song.py が要求するスキーマ。`artist` は `"cowrite"`、`song_no` は 0):

```json
{
  "artist": "cowrite",
  "song_no": 0,
  "title": "<本命タイトル>",
  "style": "<裏タグ (S3 で確定)>",
  "style_with_persona": null,
  "lyrics_display": "<全セクションの作品版を構成順に。ルビ注釈つき>",
  "lyrics_suno": "",
  "model": "<suno-spec のモデル>",
  "language": "ja",
  "vocal_gender": "<Male / Female>",
  "weirdness": 20,
  "style_influence": 60,
  "instrumental": false,
  "exclude_styles": "<少数精鋭>",
  "persona": { "use": false }
}
```

タイトルは write_core (11) に従い、S1 の仮タイトルに縛られず本命を決める (総括型・象徴物型も検討)。決めたら台帳と正データを更新。

**S6 の起点 — 台帳から `lyrics_display` を組み直す**: 台帳の「セクション作品版」の各確定セクションを構成表の順に連結し、正データ (.json) の `lyrics_display` に (再)生成する。以降のスクリプトはこの .json だけを読む。台帳が真実源・.json は毎回ここで組み直す運用なので、途中で両者がズレていても S6 のこの一手で必ず吸収される。

```bash
JSON="<ARTIST_ROOT>/.production/cowrite_<slug>.json"
# 入稿版生成 (D39): 日本語曲は lyrics_display から lyrics_suno を機械変換して書き戻す
#   (英語など変換不要な言語では実行せず、lyrics_suno に lyrics_display をそのままコピーする)
python3 "<プラグインルート>/scripts/build_submission_lyrics.py" --json "$JSON" --report
# 機械検証
python3 "<プラグインルート>/scripts/validate_song.py" "$JSON"
```

- **入稿版生成の残存漢字**警告 (`--report`) が出たら、台帳の該当行に読み注釈を足して `lyrics_display` を組み直し、再変換する
- `--report` の**助詞ローマ字化リストは必ず目視する** (これが唯一のガード。validate_song.py はこの偽陽性を検出せず PASS を返すので、飛ばすと壊れた語のまま完成扱いになる)。語中の は/へ/を の誤変換 (例: 部屋(へや) → `e や`、母(はは) → `は wa`、自販機(じはんき) → `じ wa んき`) を拾い、該当語をカタカナ表記にして作品版を直し再変換する (助詞ヒューリスティックの設計限界)
- **FAIL** → 原因の工程 (Style 系 / 歌詞系) を直し、入稿版を再生成して PASS まで提示しない。**WARN** → 判断して提示文に添える (行密度 WARN が早口セクションを指したら S5 に一瞬戻って該当セクションを詰め直す選択肢を P に出す)
- **オリジナリティ**: cowrite が最終著者。write_core (08) の 3 観点 (フレーズ / 構造 / タイトル) で自己検査し、タイトル・実在名を確認する (Web 照合は無い。第 1 層規則と自己検査で担保)

PASS したら**入稿セット**を提示する。テンプレートは oneshot と同じ (suno.com Custom Mode の欄ごとに貼れる形式):

````markdown

```

## Exclude Styles
````

## S7 — 完成後 (テイク報告・部分修正)

oneshot の完成後の作法に合わせる。

- P が**テイク報告**をくれたら受け止める (「1 テイク目が良かった」等)。数字の話は真顔で
- **部分修正**の依頼 (「2 番 A メロが早口」「サビもう少しキャッチーに」) は、S5 の 1 セクション手順に戻って**該当セクションだけ**直す (他は凍結)。直したら S6 の機械検証を通し直して入稿セットを提示し直す
- スタイルだけの修正 (「もっと激しく」) は正データの `style` / スライダーを直して再検証する
- 修正のリトライは 1 回ずつ小さく回す (症状を 1 つずつ潰す。06 第 3 部の運用ルール)
- P が「完成」と言ったら台帳の状態を「完成」にし、state.md を締める。作業ファイル (.md / .json) は `.production/` に残す (次の修正依頼で使える)
